学习GitHub（非命令行）操作总结之二

发表于 2015-08-02   |  

学习GitHub（非命令行）操作总结的第二篇。这篇文章的内容是对GitHub操作背后原理的介绍，算是很重要的内容。

简单分支操作

分支的概念

我们对某个仓库的具体内容的操作，比如对文件的增删等等，在提交之后都会形成一个个提交版本，每一个提交均会包含指向前一次提交的“指针”，这样就会形成前面出现过的一长串的提交链。

而这一长串的提交就放在分支中，在图片中显示就是有一个名为“master”的指针指向最后一次提交，这里一般来说master是GitHub默认指针名，也就是默认分支。

这样，出现了分支名指针的一长串提交就成为了分支。

master分支是默认的分支，中文名为主分支，它和用户自己创建的其他分支在原理上没有区别。而GitHub官方建议是master分支应该用来存放稳定版本的代码。

master 分支上的所有代码都应该是可以部署的

创建新分支

首先进入“Branches”一栏，点击master分支右边的“+”，输入新建分支的名字，然后点击“Branch”，就会创建一个新分支，注意这个新分支的内容是和master分支完全一致的。

点击新分支“anomaster”右边的“Switch to This Branch”可以进行切换分支。

然后来看背后是怎么实现的。新分支和master分支内容一样，并不意味着创建了master的副本，而是创建了一个名为anomaster的新指针和master指针指向了同一次提交，没有创建master的副本。

如果在切换到anomaster分支的情况下作出了修改，并进行提交，则master分支指针指向的提交没变，而anomaster分支指针指向了c3。

创建新分支的目的

前面说到过，master分支上的代码是十分重要的，但如果我们对于某个项目有了新的想法怎么办，那就可以创建新分支，切换到这个新分支后来进行实践，这样就不会污染到master分支上的重要代码。如果新的思路很好就可以合并到master分支，也就是下一节要介绍的内容。

实际操作

首先切换到anomaster分支，然后打开编辑器，添加一个新文件anomaster，内容为“this is a new branch named anomaster”

然后打开客户端，进入“Changes”栏，可以发现只有一个文件anomaster，说明这是anomaster分支，然后提交。

进入“History”栏查看可以发现所有提交，不管是master分支还是anomaster分支，这点也恰好说明了前面新建指针理论的正确。

这时候我们新建的分支还没有同步到远端的仓库，进入“Branches”栏，点击“Publish”就可以将新分支发布到远端仓库，完成后变为“Published”。

再回到网页端，可以发现branches数变为了2

删除分支

当我们发现某个分支已经没有用时，我们会选择删除它，这里需要注意的是当前分支是无法删除的，不在使用的分支才允许被删除，这样的删除不仅会删除本地的分支也会删除掉远端的分支。不过master分支就算不在使用也是无法删除的，因为它是默认分支。

当前分支删除选项灰白，无法删除。

非当前分支删除选项可以使用。

如果只想删除远端的分支，不想删除本地的分支的话，可以选择选项框中的“Unpublish”选项。

再回到网页端就会发现anomaster分支已经不存在了。

网页端创建新分支

图中的文本框有两个作用，一是当分支较多时查找分支，二是创建新分支，这里创建一个新分支testbran

在客户端点一下“Sync”就可以将远端创建的新分支同步到本地客户端。

如何修改默认分支

前面提到过默认分支master是无法删除的，如果想把它删除就只有修改默认分支。进入网页端，点击右边的“Settings”。

点击“Default branch”进行选择。

这样就修改了默认分支为testbran。

不过一般来说是不会去修改默认分支的，也不会去删除master分支。

分支合并

当新的分支被认可时，就可以将新分支合并到master分支中，这里我决定合并anomaster分支和master分支。

先在客户端切换到master分支，然后点击右上角的“Merge View”，就会出现两个融合框，Merge本身就是就是合并、融合的意思，这里是将前一个框中的分支合并到后一个框中。

客户端比较好的一点就是这里可以将对应的分支拖动到融合框中，拖动完成后点击“Merge Branches”。

再回到“History”栏，可以看到在原有master分支基础上增加了两个提交：new branch和Merge branch ‘anomaster’，毫无疑问，new branch是anomaster分支中一次提交的内容，而另外一次提交则是特殊的融合提交。

这个特殊的融合提交就相当于图中的c5，它不会修改内容，作用就是将两个分支合并起来，它拥有两个指针，一个指向anomaster分支的最新提交，另外一个指向master分支的最新提交，最后master指针移动到c5，这样master分支就代表原有两分支的合并。这时候anomaster分支可以删除了。

分支合并的方式

分支合并除了上面提到的merge之外，还有另外一种叫做rebase（变基），这两者的作用都是一样的，就是将两个分支进行合并，但产生的后果不同。

在某个分支B上对分支A进行merge后，分支A就被合并到分支B上去了，但分支A还存在，而且在合并后的分支会产生一个“Merge XXX”提交。

在某个分支B上对分支A进行rebase后，分支A就被合并到分支B上去，但分支A不存在了，只剩下了一条分支，在合并后的分支不会产生“Merge XXX”的提交。

举个例子，远端仓库别人在master分支进行了提交，这时候本地的master分支还没有那个提交，需要进行同步来将远端和本地两处的master分支进行合并（注意这里远端和本地的master分支严格意义上是不同的两个分支，所以可以进行合并），合并完成后可以发现commit的历史版本中并没有“Merge XXX”的提交，因为这种合并是rebase合并。

代码冲突（Sync Conflicts）

在实际的开发中，一个仓库的不同分支可能都在同时开发，当代码开发位置不冲突时可以直接合并（merge或rebase），这样不会出现什么问题。

但当代码开发位置发生冲突时，比如都在编辑同一个地方，这个时候合并（merge或rebase）就会产生问题，因为git它不知道该听谁的编辑，这个时候就只能交给人来解决。

比如两个人均要修改邮件地址，但这两个人却修改成了不同的邮件地址，当某一方提交后，后面一点的再提交就会出现同步错误的提示，这个时候打开编辑器可以发现冲突位置的变化。

<<<<<<< Head代表的是当前分支，这里是本地的master分支，=====是两个代码冲突块的分界线，下面的>>>>>>> origin/master表示的是远端的master分支，解决冲突就是把上面的三行“冲突标示符”都删掉，然后修改代码。之后，回到客户端，点击“Commit to Master”。 这样，这次分支合并就完成了，也会生成一个“merge commit”。
