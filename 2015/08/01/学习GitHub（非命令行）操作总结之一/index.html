学习GitHub（非命令行）操作总结之一

发表于 2015-08-01   |  

对于GitHub的操作一直不是很熟练，无论是命令行还是GUI。下了一本《Pro Git》的电子书，却一直没静下心来看，在慕课网上发现了一门课程：版本控制入门 – 搬进 Github，主要是从网页操作和客户端操作的角度来讲Git的操作的，而且也对Git后面的原理做了介绍，相比于《Pro Git》中命令行的教学更为直接也方便上手。本篇文章是我对该门课程学习的一个总结（学习笔记），主要是网页端的操作和Mac平台GitHub客户端的操作，其间也有从其他资料找到的信息。

浏览器中使用GitHub

浏览器中创建一个仓库的过程

首先点击上面“网页端”的超链接，会出现这样一个页面

关于注册的相关过程很简单，这里就略过。

注册之后再进入GitHub首页就会进入个人账户，找到如图所示的地方

点击写有“+ New repository”的绿色按钮，会出现如下界面

在“Repository name”中填写仓库的名字，这里是“demo”；“Description”正如后面括号写的，这里是可选的，主要是对该仓库内容的描述；再下面的“Public”和“Private”对应了两种仓库类型，只有付费账户才允许创建，区别主要是前一个别人可以访问而后一个不允许；“Initialize this repository with a README”选中则会创建一个markdown文档，也是对该仓库内容的介绍，不过相比前面的“Description”可以自定义地更为详尽，下面那两个“Add .gitignore”和“Add a license”暂时不懂。最后点击“Create repository”的按钮。

这就是最终一个仓库创建后的初始界面

Repository

不同于其他软件的项目（project）或者应用（application）等，GitHub使用的是仓库（repository）

Commit

Commit是GitHub中极为常见的一个词，虽然慕课网上那个课程老师称之“做一个版本”或者叫“版本”，但我在看了其他资料后，感觉还是称其为“提交”更为顺口，当然那位老师的说法也不是错误，只是另外角度的阐释，让我对commit这个词有了更深入的理解，相当于每一次提交操作（增加内容、删除内容、修改内容等）就等于是产生了一个新的版本信息。

这里来增加一个文件“test”。点击后面那个“+”，进入编辑页面。

点击在上方的“Preview”可以对文件内容进行预览。

如果想获得更好的编辑输入体验，在同一水平的右边有一个“Edit in fullscreen”的小图标进入全屏模式，有意思的是这个模式还叫做“Zen Mode”，实在是逼格满满（笑

最下方的提交框的两个文本框均为该次提交的留言，前一个简略，后一个详细；这里先选中“Commit directly to the master branch”，后一个暂时不懂。然后点击“Commit new file”。

可以看到除了原有的README文件外多出了一个test文件，最重要的是“commits”数已经变成了2！说明产生了一个增加文件的版本信息，点击“commits”可以进入显示该仓库所有提交的页面。

点击某次提交进入该次提交具体页面。

可以看出是“shuaiyunzhang”（who）在8分钟前（when）做了“Showing 1 changed file with 1 addition and 0 deletions.”的修改（what）。

另外commit的版本号是图中的“d8768678e07c190395bb6cd1f84f886d622a207e”，四十位的16进制数，在版本号commit的左边parent是该次提交的前一次提交的版本号的前7位（为了显示方便）。

这是提交的一个示意图，表示的是所有提交的历史线，比如c1代表了第一次提交，版本号为右边那一长串数字，c2代表第二次提交，以此类推。

GitHub客户端的使用

客户端中创建一个仓库的过程

相比于浏览器，大家应该更习惯于在本地也就是客户端进行操作，使用自己喜欢的编辑器，也更为高效。

首先点击左上角的“+”号，会出现三个选项，分别是“Add（添加本地仓库）”、Create（创建新仓库）“和”Clone（克隆远程的仓库）“，这里我是选择了克隆前面创建的demo仓库。

然后在本地找到该仓库的位置，并用文本编辑器打开，我这里是SublimeText。

在test文件中添加新的一行“this is a new line”，然后保存，这时候就会发现客户端出现了变化，在“Changes”一栏中显示“1 Changes”，test前面的小勾表示被选中，右边的文件具体内容左边的蓝色区域也代表选中。

所以如果点击一下左边的蓝色区域，就可以取消选中，如下图

当然这里我最后还是选上了，在下方的两个文本输入框填写相应信息就可以提交到master分支了（这个和浏览器的操作一致）

然后点击上方的“History”栏来查看每次提交（commit）的信息。

从“History”就可以看出每次提交的具体信息，另外如果你认为哪次提交的信息出现了错误，想撤销这次提交，就可以点上上张图片里的“Undo”来撤销该次提交，不过需要注意的是：Undo只适合没有同步（Sync）的提交，如果已经将该次提交同步了，则Undo无效。

当然想要撤销掉已经同步的提交也是有办法的，那就是点击下图的“Revert this Commit”

再来查看“History”中每次提交信息，可以发现出现了一个“Revert XXX”的提交，作用是上次提交添加了什么，名为“Revert XXX”的提交就撤销掉

如果感觉很多个提交均有问题，想把那些认为不好的提交全部撤销怎么办？那就可以先选中自己认为比较好的提交，然后选择“Roll Back to this Commit”

可以发现，回滚也是提交了一个新提交，名为“Revert XXX”，抵消掉了前几次提交的操作

同步

同步是GitHub中很重要的一个操作，因为你的代码终究是会和别人分享的，所以需要同步。我在实际操作时发现了一个问题，那就是在慕课网的视频上右上角是“Publish”的字样，而我这里是“Sync”的字样，我想原因应该是我这里是克隆，视频上是创建的原因，这里我直接点击Sync就行了，没有再弹出一个文本框填东西。

当我们完成了同步操作再回到浏览器页面，可以发现commits数已经变为了6
